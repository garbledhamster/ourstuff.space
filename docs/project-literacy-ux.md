# Project Literacy UX

## Purpose

This dashboard should teach users how to read their own work. It should explain the meaning of activity, projects, warnings, tools, and next actions without turning the first screen into a manual.

## Teaching Mode Rules

- Keep concise mode as the default scanning surface.
- In teaching mode, add one or two sentence explainers beside the surface being read.
- Prefer three prompts: "What am I seeing?", "Why it matters", and "Next useful action".
- Use concrete nouns from `UBIQUITOUS_LANGUAGE.md`.
- Avoid shame, ranking, diagnosis, and productivity pressure.
- Explain acronyms before using them in visible copy.

## Copy Patterns

Use:

- "This snapshot reads..."
- "This matters because..."
- "Next useful action..."
- "If this looks empty..."
- "Refresh only when you expect new cloud or backend state."

Avoid:

- "Optimize"
- "Boost engagement"
- "Critical failure" for ordinary empty states
- "API" without context
- "Needs attention" without a reason

## Dashboard Surface Guidance

Daily Return:

- What am I seeing? Four paths into the user's private workspace.
- Why it matters: The user does not need to choose the whole app, only one path.
- Next useful action: Pick the path that matches what needs attention today.

Balance:

- What am I seeing? A snapshot of recent activity across Mind, Body, Spirit, and Life.
- Why it matters: The chart shows where records have been landing, not whether the user is doing well.
- Next useful action: Open the quietest or loudest area and add one useful note.

Life Tools:

- What am I seeing? Calendar, tasks, and projects are separate ways to read the same Life work.
- Why it matters: A project teaches structure, while a task teaches the next concrete action.
- Next useful action: If work feels vague, create a project. If work is clear, create a task.

Warnings and Empty States:

- Always name the missing thing.
- Always explain why it matters.
- Always offer a small next action.
